# Техническое задание для Cursor  
**Проект:** AI Flower Assistant  
**Тип продукта:** веб-приложение на React  
**Приоритет:** mobile-first, минималистичный интерфейс, архитектура с запасом на масштабирование

---

## 1. Назначение документа
Этот документ является базовым техническим заданием для генерации и разработки проекта в Cursor.  
Используй его как основной источник требований к структуре, архитектуре, экранам, логике интерфейса и базовому поведению приложения.

Если какие-то детали не описаны явно:
- не усложняй проект сверх MVP;
- сохраняй чистую и масштабируемую архитектуру;
- делай упор на мобильную версию;
- пиши код на TypeScript;
- используй переиспользуемые компоненты;
- закладывай удобное подключение backend API.

---

## 2. Краткое описание проекта
Приложение позволяет пользователю общаться с AI-ассистентом в формате чата.

Пользователь может:
- задать текстовый вопрос;
- прикрепить фотографию цветка;
- получить ответ от AI по фото и/или тексту.

AI должен уметь:
- определить вероятное название цветка;
- показать вероятность / степень уверенности;
- указать возможную проблему растения, если она есть;
- предложить решение или рекомендацию;
- отвечать на обычные текстовые вопросы по растениям и цветам.

Также в приложении должны быть:
- стартовая страница;
- основной экран чата;
- история чатов;
- личный кабинет (простая версия);
- база знаний.

---

## 3. Цель MVP
Создать первую рабочую версию веб-приложения, в которой пользователь сможет:
1. открыть сайт;
2. перейти со стартового экрана в чат;
3. задать вопрос текстом;
4. загрузить фото цветка;
5. получить AI-ответ;
6. открыть историю чатов;
7. перейти в базу знаний;
8. открыть простой личный кабинет.

---

## 4. Что входит в MVP
### Входит
- стартовая страница;
- экран чата;
- отправка текстовых сообщений;
- загрузка изображения в чат;
- вывод ответа AI;
- история чатов;
- страница базы знаний;
- страница личного кабинета;
- адаптивность с приоритетом под мобильные устройства;
- подготовленная архитектура проекта под дальнейшее развитие.

### Не входит в MVP
- сложная авторизация;
- подписки и платежи;
- роли пользователей;
- админ-панель;
- сложная аналитика;
- пуш-уведомления;
- мультиязычность;
- real-time синхронизация между устройствами.

## 5. Основной пользовательский путь
### Сценарий 1. Первый вход
1. Пользователь открывает сайт.
2. Видит стартовую страницу с кратким описанием сервиса.
3. Нажимает кнопку **«Начать»**.
4. Переходит на основной экран чата.

### Сценарий 2. Текстовый вопрос
1. Пользователь вводит вопрос в поле ввода.
2. Нажимает кнопку отправки.
3. Видит сообщение пользователя в чате.
4. Получает ответ AI.

### Сценарий 3. Вопрос с изображением
1. Пользователь открывает чат.
2. Прикрепляет изображение цветка.
3. При желании добавляет текстовый комментарий.
4. Отправляет сообщение.
5. Получает ответ AI с анализом изображения.

### Сценарий 4. Работа с историей
1. Пользователь открывает левую панель с историей чатов.
2. Видит список прошлых диалогов.
3. Выбирает нужный чат.
4. Открывает его и продолжает переписку.

### Сценарий 5. Переход в базу знаний
1. Пользователь нажимает кнопку перехода в базу знаний.
2. Открывает страницу со статьями / карточками материалов.
3. Выбирает материал и читает его.

### Сценарий 6. Личный кабинет
1. Пользователь нажимает на аватар в правом верхнем углу.
2. Открывает страницу или выпадающее меню личного кабинета.
3. Видит базовую информацию о профиле.

## 6. Основные экраны
## 6.1. Стартовая страница
### Назначение
Первое знакомство с сервисом.

### Состав
- логотип / название проекта;
- короткий заголовок;
- краткое описание пользы сервиса;
- кнопка **«Начать»**.

### Требования
- экран должен быть простым и чистым;
- главный фокус — на одном CTA;
- дизайн должен хорошо смотреться на мобильном экране.

---

## 6.2. Экран чата
### Назначение
Основной рабочий экран приложения.

### Структура
- верхняя панель;
- кнопка открытия истории чатов слева;
- аватар / кнопка личного кабинета справа;
- центральная область сообщений;
- нижняя область ввода сообщения;
- кнопка прикрепления фото;
- кнопка отправки;
- кнопка / ссылка перехода в базу знаний.

### Логика
- экран визуально похож на ChatGPT;
- сообщения пользователя и AI должны быть визуально разделены;
- чат должен поддерживать прокрутку;
- при отправке сообщения показывать состояние загрузки;
- пользователь должен видеть прикрепленное изображение до отправки;
- после отправки изображение должно отображаться в сообщении.

---

## 6.3. Панель истории чатов
### Назначение
Быстрый доступ к прошлым диалогам.

### Состав
- список чатов;
- название чата;
- дата / время последнего сообщения;
- кнопка создания нового чата.

### Требования
- на мобильной версии панель открывается как drawer / боковая панель;
- на десктопе может быть постоянной или раскрываемой;
- пользователь может выбрать существующий чат и открыть его.

---

## 6.4. Личный кабинет
### Назначение
Базовый профиль пользователя.

### MVP-состав
- имя пользователя;
- email или условный идентификатор;
- аватар;
- кнопка выхода;
- блок с базовой информацией.

### Требования
- пока без сложных настроек;
- лаконичный экран;
- архитектурно предусмотреть расширение в будущем.

---

## 6.5. База знаний
### Назначение
Раздел со справочной информацией по цветам и уходу за растениями.

### MVP-состав
- список статей / карточек;
- страница материала;
- категории или простая фильтрация (по возможности без усложнения).

### Примеры материалов
- уход за комнатными цветами;
- частые болезни растений;
- почему желтеют листья;
- базовые рекомендации по поливу и свету.

## 7. Логика AI-ответа
Если пользователь отправляет изображение цветка, интерфейс должен быть готов к получению следующей структуры ответа от backend / AI:

```ts
type FlowerAnalysisResponse = {
  flowerName: string;
  confidence: number; // 0..1
  issue?: string;
  solution?: string;
  answerText: string;
};
```

### Что нужно показать в UI
- название цветка;
- вероятность / уверенность;
- проблему, если она определена;
- решение / рекомендацию;
- основной текст ответа.

Если пользователь отправляет только текст, AI возвращает обычный текстовый ответ.

- минимализм;

## 10. Архитектурный подход
Для проекта рекомендуется использовать **feature-sliced / модульную архитектуру**, чтобы приложение было удобно масштабировать.

### Основной принцип
Разделять код по ответственности:
- `app` — инициализация приложения, роутинг, провайдеры, глобальные стили;
- `pages` — страницы;
- `widgets` — крупные блоки экранов;
- `features` — пользовательские действия;
- `entities` — бизнес-сущности;
- `shared` — переиспользуемые компоненты, утилиты, константы, API-клиент.

## 12. Правила организации кода
### Обязательно
- один компонент — одна ответственность;
- не хранить крупную бизнес-логику прямо в JSX;
- все API-запросы вынести в отдельный слой;
- типы вынести в `types.ts`;
- роуты и константы вынести отдельно;
- переиспользуемый UI складывать в `shared/ui`;
- не допускать хаотичного импорта между слоями.

### Желательно
- использовать index.ts для удобных публичных экспортов;
- разделять `ui`, `model`, `api`, `lib`;
- использовать alias imports;
- предусмотреть моковые данные для быстрой локальной разработки.

## 15. API-слой
На фронтенде должен быть подготовлен отдельный API-слой.  
Даже если на первом этапе backend отсутствует, структура должна быть такой, чтобы легко заменить mock на реальные запросы.

### Пример endpoints
```ts
GET    /api/chats
POST   /api/chats
GET    /api/chats/:id/messages
POST   /api/chats/:id/messages
POST   /api/flowers/analyze
GET    /api/articles
GET    /api/articles/:id
GET    /api/profile
```

### Требования к API-интеграции
- централизованный API client;
- единая обработка ошибок;
- типизация ответов;
- возможность использовать mock mode;
- отдельные файлы для endpoint-модулей.
